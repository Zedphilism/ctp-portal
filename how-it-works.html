CT&P inspection operations

How this system works, end to end.

This page explains the actual operating model used in the repo: applicants submit Google Forms, records are stored in Google Sheets, supervisors assign inspectors through the web app, inspectors execute and update jobs, and dashboards provide live monitoring, schedule, analytics, map, and workload visibility.

Two operating lanes, one core pattern.

The project supports Sel Teknikal and Sel Non Teknikal. They use separate dashboards, web app variants, Apps Script endpoints, and sheet sources, but both follow the same assignment and inspection lifecycle.

Operating lanes
Sel Teknikal Technical inspections, technical dashboard, technical Apps Script backend.
Sel Non Teknikal Non-technical inspections, separate dashboard theme and separate backend endpoint.
System backbone
  • 1
    Google Forms + Sheets Requests land in sheets such as Permohonan, WDP, FAT, and the inspector roster lives in Pemeriksa.
  • 2
    Apps Script backend Processes assignments, conflict checks, notifications, evidence, document generation, and status refresh.
  • 3
    Dashboards + Web App Dashboards are for monitoring. The web app is for operational work, assignment, and inspector execution.

System components

Each part of the portal has a clear role. The landing page selects the working lane, the dashboard summarizes and monitors, and the web app is where assignments, execution updates, evidence, and documents are handled.

Separate script URLs are configured for technical and non-technical operations, so each lane can read from its own data source while keeping a familiar user experience.

End-to-end workflow

This is the practical sequence the system follows today, based on the implemented frontend and backend logic in the repository.

1

Request is submitted

A user fills in a Google Form for the relevant inspection stream. The response lands in the corresponding Google Sheet row.

Google Form -> Sheet row
2

Supervisor is alerted

The backend can send a submission email so supervisors know a new request is waiting and should be assigned.

sendSubmissionEmail()
3

Web app loads the live workspace

The web app requests pending jobs, available inspectors, full roster, and active schedule data from Apps Script.

getWebAppData()
4

Supervisor assigns the job batch

The supervisor selects jobs, enters start and end dates, chooses the lead inspector and optional second inspector, then runs assignment.

processAssignment()
5

System writes back and notifies

On success, the job rows are updated to Assigned, inspector names and dates are stored, roster status is refreshed, and an assignment email is sent.

sendAssignmentNotification()
6

Inspector executes and reports

Inspectors view their assigned jobs, update job status, upload evidence, and generate official task documents while the dashboard tracks progress.

updateInspectorJob()

Best for overview, monitoring, analytics, and management visibility. It answers: what is happening, where, how many, and who is loaded.

Inspector assignment logic

The assignment flow is implemented in the supervisor web app and backed by Apps Script. Assignment is not just a UI action: it validates date conflicts, updates inspector availability, logs the action, and triggers notification email delivery.

1

Pending jobs are loaded from the request sheets

The supervisor workspace pulls pending jobs from Permohonan, WDP, and FAT. Jobs are treated as pending when status is blank or contains Pending.

2

Supervisor selects jobs, dates, and inspectors

Step 1 is job selection. Step 2 captures the assignment window, lead inspector, and optional second inspector. A lead inspector and start date are required.

3

Conflict detection runs before saving

getInspectorConflicts_() compares the requested date window against all active jobs already associated with the same inspector. If any overlap exists, the assignment is blocked.

4

Successful assignment updates the source data

The backend writes Assigned into the job row, saves start and end dates, stores both assigned inspectors, logs the event, and gathers unit and location details for the outgoing email.

5

Inspector availability is refreshed

If the start date is today or within the active date range, the inspector can become Sedang Bertugas. Future assignments set Telah Ditugaskan. Otherwise the inspector returns to Aktif.

6

Execution updates keep the lifecycle moving

Inspectors later update their job to statuses such as In Progress or Completed, and the backend can save structured inspection notes and upload evidence.

Status flow

The operational lifecycle typically moves from Pending to Assigned to In Progress, then to Completed or Cancelled.

Schedule scope

The schedule view includes jobs that are Assigned, In Progress, or Completed, so supervisors can see the active and recent date footprint.

Reassignment

Reassigning a job reruns conflict checks, updates the inspector names and dates, and refreshes status for both the old and new inspectors.

Cancellation

Cancelling a job sets the row to Cancelled and refreshes inspector availability so the roster no longer shows a stale assignment.

Inspector reports

Execution updates can store structured JSON containing a brief report, extracted inspection date, inspector name, and submission timestamp.

Notification target

Assignment email recipients are resolved from the inspector roster sheet by matching the inspector name to the stored email address.

Outputs, documents, and supporting features

Beyond assignment itself, the system also supports inspector execution, evidence handling, document generation, and reference browsing. These pieces turn the portal from a tracker into an operational workflow tool.

Docs

Borang Tugas Rasmi

The Dokumen section can generate the official task document for the selected inspector and assigned job set using Drive-backed output folders.

Drive

Evidence upload

Inspectors can upload evidence files from the execution panel. The backend accepts base64 upload payloads and stores them with the job context.

Ref

Reference library

The Reference tab helps the team browse uploaded surat permohonan documents by month, applicant, unit, or phone number.

Plan

Schedule and roster

Schedule shows the date footprint of active work, while roster highlights who is active, assigned, or currently on duty.

Exec

Inspector execution panel

Each inspector can review only their assigned jobs, choose a focused job, submit status changes, and continue evidence and report work from one place.

AI

AI Copilot

For authorized accounts, the supervisor workspace can generate cluster suggestions and WhatsApp-ready assignment text from the current selection.
